## 1. Паттерн «Цепочка обязанностей»

### В: Что такое паттерн «Цепочка обязанностей»? Зачем он нужен?
Поведенческий паттерн. У запроса есть несколько потенциальных обработчиков, выстроенных в цепочку. Каждый обработчик решает: могу я обработать этот запрос? Если да — обрабатывает; если нет — передаёт следующему. Отправитель не знает, кто именно из цепочки возьмёт запрос. Цепочка может быть линейной (список) или древовидной (как у нас).

Зачем нужен: отвязать отправителя от обработчиков. Можно добавлять/убирать обработчиков без изменения отправителя. Каждый обработчик не знает про остальных — только про «следующего».

### В: Какая у тебя цепочка?
У меня цепочкой служит дерево фигур из Практики 1: корень `Fraction` → числитель `Integral(x, dx)` и знаменатель `Matrix` → 9 ячеек-`Atom`-ов. У каждого узла есть указатель на родителя `parent_`. Когда приходит запрос с координатой `(x, y)`, он стартует с листьев и идёт ВВЕРХ по `parent_` — первый узел, в чей bbox попадает точка, обрабатывает запрос.

### В: Где у тебя именно «цепочка»?
Метод `Formula::bubbleUp(t, x, y)`:
```cpp
bool bubbleUp(RequestType t, int x, int y) const {
    if (contains(x, y)) { doHandle(t, x, y); return true; }
    if (parent_) return parent_->bubbleUp(t, x, y);
    return false;
}
```
Узел проверяет себя; если попал — обрабатывает, если нет — передаёт родителю. Если `parent_ == nullptr` (корень) и не попал — `false` (запрос «вышел из цепочки»).

### В: Почему рассылка начинается с листьев?
Так Сергей подсказывал и так каноничнее по GoF: запросы посылаются «от потомков к родителям». Листья — это визуально самые верхние фигуры на экране (мелкие, внутри контейнеров). Поэтому логично, что если точка попала в лист, его обрабатывает он; а если не попала ни в один лист — поднимаемся выше по дереву и ищем подходящий контейнер.

### В: Как у тебя устроена «рассылка с листьев»? Покажи код.
Функция `dispatch()`:
```cpp
void dispatch(const Formula* root, RequestType t, int x, int y) {
    std::vector<const Formula*> leaves;
    collectLeaves(root, leaves);

    // ФАЗА 1: есть ли лист, который сам содержит точку?
    for (const Formula* leaf : leaves) {
        if (leaf->contains(x, y)) { leaf->bubbleUp(t, x, y); return; }
    }

    // ФАЗА 2: ни один лист не попал — поднимаемся по parent_ от каждого
    //         листа и берём самого глубокого предка, содержащего точку.
    const Formula* best = nullptr;
    int bestDepth = -1;
    for (const Formula* leaf : leaves) {
        const Formula* h = leaf->parent() ? leaf->parent()->findHandler(x, y) : nullptr;
        if (h) {
            int d = depthOf(h);
            if (d > bestDepth) { best = h; bestDepth = d; }
        }
    }
    if (best) best->bubbleUp(t, x, y);
    else std::cout << "запрос потерян\n";
}
```

### В: Почему две фазы? Зачем `findHandler` если есть `bubbleUp`?
Если бы я наивно сделала `for each leaf: leaf->bubbleUp(...)` — получился бы баг. Пример: клик в Matrix между ячейками. Первый лист — Atom `x` из Integral, он не содержит точку, поднимается до Integral, тот тоже не содержит, поднимается до Fraction — Fraction содержит, и Fraction обрабатывает. А правильно — Matrix должна обработать, т.к. она «глубже» в дереве и «выше» на экране.

Чтобы это починить:
- В ФАЗЕ 1 я проверяю каждый лист на прямое попадание — если есть, обрабатывает он (он самый глубокий).
- В ФАЗЕ 2 (если ни один лист не попал) я обхожу цепочки от ВСЕХ листьев и ищу самого глубокого предка через `findHandler`. `findHandler` — это `bubbleUp` без побочных эффектов (не обрабатывает, просто возвращает узел). Потом обрабатывает один-единственный — самый глубокий.

### В: Что значит «глубокий»? Как меряешь?
Функция `depthOf(node)` идёт от узла вверх по `parent_` и считает шаги. У корня глубина 1, у его прямого ребёнка — 2, и т.д. Самый глубокий = самый дальний от корня = самый мелкий по геометрии = «верхний на экране».

### В: Как ты выставила `parent_`?
В конструкторе каждого композита и в `Matrix::setCell`. Например, в `Fraction`:
```cpp
Fraction(FormulaPtr top, FormulaPtr bottom)
    : top_(std::move(top)), bottom_(std::move(bottom)) {
    top_->setParent(this);
    bottom_->setParent(this);
}
```
И в матрице:
```cpp
void setCell(int r, int c, FormulaPtr child) {
    child->setParent(this);
    cells_[r][c] = std::move(child);
}
```
После того как дерево собрано в `main()`, у всех узлов кроме корня `parent_` указывает на их родителя.

### В: Что такое «функция диспетчеризации» из примечания 2?
В методичке сказано: «для обработки нескольких типов запросов написать функцию диспетчеризации для идентификации типа запроса». У меня это метод `Formula::doHandle(RequestType t, ...)`:
```cpp
if (t == RequestType::Help) { ... консольный вывод ... }
else { ... запись в print.log ... }
```
По типу запроса (`Print` или `Help`) выбирается своя ветка обработки. Это «функция диспетчеризации».

Тип запроса описан `enum class RequestType { Print, Help }`. `enum class` — строгий enum: его нельзя случайно сравнить с int, имена пишутся через `RequestType::Print`.

## 2. Реализация

### В: Как формула узнаёт свой прямоугольник на экране?
В базовом классе `Formula` есть **mutable-поля** `x_, y_, w_, h_`. В каждой реализации `draw(out, x, y)` первой строкой:
```cpp
x_ = x; y_ = y; w_ = me.w; h_ = me.h;
```
То есть при отрисовке каждая формула «фиксирует» свою позицию. После того как корневой `draw()` отработал, у ВСЕХ узлов дерева заполнены `x_, y_, w_, h_`. Потом метод `contains(px, py)` просто сравнивает координаты с этим прямоугольником.

### В: Что такое `mutable`?
`mutable` — модификатор поля, разрешающий менять его даже у const-объекта. У меня `draw()` помечен как `const` (по смыслу он «только рисует»), но при этом ему нужно ЗАПИСАТЬ свои координаты. Без `mutable` компилятор не разрешил бы. Это стандартный приём для «логически константных» методов с кэшем/счётчиком.

### В: Зачем `children()`? Что он возвращает?
`children()` — виртуальный метод базы `Formula`. Возвращает `std::vector<const Formula*>` — указатели на дочерние формулы. Атомы не переопределяют, у них дети пустые. Композиты возвращают свои поля:
- `Fraction` → `{top_.get(), bottom_.get()}`
- `Integral` → `{body_.get(), diff_.get()}`
- `Matrix` → все 9 ячеек

В моём коде `children()` нужен для `collectLeaves()` (рекурсивно собрать все листья дерева) и для `printBoxes()` (распечатать дерево с bbox для отладки). Для самой цепочки `children()` не нужен — она идёт через `parent_`.

### В: Почему `.get()` у unique_ptr?
`.get()` возвращает СЫРОЙ указатель, не отдавая владение. Это нужно потому, что `unique_ptr<Formula>` нельзя копировать (он уникальный), но мы хотим только «посмотреть» на ребёнка через указатель. `Formula*` достаточно для вызова виртуальных методов.

### В: Покажи, как HandlePrint работает на координате внутри Atom 'a'
Допустим, Atom 'a' имеет bbox `(48, 36, 12, 26)` — клик `p 50 50`.
1. `main()` зовёт `dispatch(root.get(), Print, 50, 50)`.
2. `collectLeaves` собирает 11 листьев (`x`, `dx`, 9 атомов матрицы).
3. ФАЗА 1: перебираем листья. Atom `a` `contains(50, 50)` → true. Зовём `a->bubbleUp(Print, 50, 50)`.
4. `bubbleUp` у Atom 'a': `contains(50, 50)` → true → обрабатывает. Печатает в `print.log` строку `[Print 50,50] Я Atom('a'), позиция (48,36), размер 12x26`. Возвращает `true`.
5. `dispatch` выходит из цикла.

### В: А если точка в Matrix, но между ячейками?
Например, клик `h 70 50` — попадает в bbox Matrix, но не в bbox конкретной ячейки.
1. ФАЗА 1: перебираем 11 листьев. Никто `contains(70, 50)` → false.
2. ФАЗА 2: для каждого листа зовём `parent()->findHandler(70, 50)`.
   - У ячейки матрицы parent = Matrix. `Matrix.findHandler` → contains? Да → возвращает Matrix. Глубина Matrix = 2.
   - У Atom `x` parent = Integral. `Integral.findHandler` → contains? Нет → parent = Fraction. `Fraction.findHandler` → contains? Да → возвращает Fraction. Глубина Fraction = 1.
3. Сравниваем по глубине: Matrix (2) глубже Fraction (1) → `best = Matrix`.
4. Зовём `Matrix->bubbleUp(Help, 70, 50)` — Matrix contains → обрабатывает.

### В: Что если точка вне всех формул?
ФАЗА 1 → ни один лист не содержит. ФАЗА 2 → ни одна цепочка `findHandler` не нашла предка с `contains == true`. `best == nullptr`. Печатается: «никто не содержит точку — запрос потерян».

Это и есть «запрос остаётся необработанным» из методички.

### В: Почему ты не зовёшь bubbleUp прямо для всех листьев в цикле?
Из-за бага, который я описала выше: листья соседних поддеревьев поднялись бы по своим цепочкам через свой ближайший контейнер до общего корня — и корень (или общий контейнер) захватил бы запрос раньше, чем мы добрались бы до нужного контейнера. Поэтому я разделила: сначала проверить, не попал ли вообще лист (ФАЗА 1), а потом — отдельно искать самого глубокого подходящего предка через `findHandler`, который не делает побочных эффектов.

## 3. C++ детали

### В: Что такое `enum class`? Чем отличается от обычного `enum`?
`enum class RequestType { Print, Help }` — это **строгий** enum (введён в C++11). Отличия от обычного:
- Имена пишутся с префиксом: `RequestType::Print` (а не просто `Print`).
- Нельзя случайно сравнить с int: `if (t == 0)` не скомпилируется.
- Имена не «утекают» в окружающее пространство имён — нет конфликтов.

Это лучше для типобезопасности.

### В: Где у тебя `mutable`? Зачем?
В базовом классе `Formula`:
```cpp
mutable double x_ = 0, y_ = 0, w_ = 0, h_ = 0;
```
Поля mutable, потому что `draw()` объявлен `const` (он «логически» только пишет в поток), но при этом меняет эти поля для хранения позиции. Альтернатива — снять const с `draw()`, но это пришлось бы менять во всех потомках и интерфейсе.

`parent_` НЕ mutable: он устанавливается один раз при сборке дерева (через не-const `setParent`) и больше не меняется.

### В: Почему `std::ofstream f("print.log", std::ios::app)`?
`std::ios::app` означает «открыть на ДОЗАПИСЬ в конец файла». Каждый вызов `Print` добавляет новую строку, не стирая старые. Без этого флага файл бы перезаписывался при каждом вызове.

### В: Что такое `static_cast<double>(text_.size())`?
`text_.size()` возвращает `std::size_t` (целочисленный unsigned). Если умножать на `fontSize_ * 0.55`, без явного приведения возможна потеря точности или неожиданное поведение. `static_cast<double>` — явное приведение к double для безопасной арифметики.

## 4. Связь с Практикой 1

### В: Что изменилось по сравнению с твоей Практикой 1?
Я дополнила базовый класс `Formula`:
1. `mutable` поля `x_, y_, w_, h_` для запоминания позиции.
2. Поле `Formula* parent_` + методы `setParent` / `parent`.
3. Виртуальные методы `typeName()` и `children()`.
4. Методы `contains()`, `describe()`, `bubbleUp()`, `findHandler()`, `doHandle()`.

Существующие классы (`Atom`, `Fraction`, `Integral`, `Matrix`) изменились так:
- Добавилось `typeName()` (одна строка).
- В `draw()` первой строкой: запоминаем `x_, y_, w_, h_`.
- Композиты переопределили `children()` со списком детей.
- В конструкторах композитов / `Matrix::setCell` добавился `setParent(this)` для каждого ребёнка.

Логика отрисовки осталась идентичной — поэтому SVG и сама формула выглядят так же, как в Практике 1.

Из дерева убрала `Indexed` (подстрочник со scale-трансформом) — у него bbox были некорректны из-за масштабирования, что мешало `contains()`. Новое дерево: `Fraction[ Integral(x, dx) / Matrix3x3 ]`, все bbox честные.

### В: Почему именно формулы, а не геометрические фигуры?
Методичка говорит «фигуры из практики №1». У меня в Практике 1 — математические формулы. Соответственно «фигуры» здесь = формулы. Все требования паттерна выполнены:
- Иерархия объектов есть (Composite-дерево).
- Каждая формула имеет прямоугольную область (bbox) с координатами в SVG → есть смысл у `contains(x, y)`.
- Цепочка работает по этому дереву через `parent_`.
